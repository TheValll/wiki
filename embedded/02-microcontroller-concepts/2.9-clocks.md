# Part 2.9 — Clocks

Without a clock, nothing in a microcontroller works. The clock is the **heartbeat** that synchronizes every operation: the CPU executing an instruction, a peripheral sampling a signal, two devices talking over a bus. Every digital circuit advances on each tick.

This page covers where the clock comes from, how it gets distributed, and the three knobs (division, gating, domains) that let you tune the system for performance and power.

---

## 2.9.1 — The Analogy

A microcontroller is a marching band. Without a beat, no one stays in step — the brass plays at one tempo, the percussion at another, the music collapses.

The **drum beat** = the clock. Everyone marches on the beat.

| Marching band concept | MCU clock concept |
|---|---|
| Drummer (the source of tempo) | **Clock source** — oscillator or PLL |
| The same beat heard by every section | **Clock tree** — distribution network |
| Brass plays half-time, percussion full-time | **Clock domains** — different parts at different rates |
| Telling the brass to stop while percussion continues | **Clock gating** — disable the clock to a part to save power |
| Slowing the whole thing for the slow song | **Clock division** — divide the source frequency for slower peripherals |

Without coordination, no music. Without coordination, no working chip.

---

## 2.9.2 — Clock sources

A clock source generates the **square wave** (regular alternating high/low signal) that drives the system. Several types coexist on a typical MCU:

| Source | Origin | Properties | Typical use |
|---|---|---|---|
| **Crystal oscillator (XTAL)** | External crystal + internal driver circuit | High accuracy (±20 ppm), stable, requires 2 pins | Main system clock (e.g. 40 MHz on ESP32-C3) |
| **RC oscillator** | Internal resistor + capacitor | Low cost, no external parts, lower accuracy | Low-power / always-on clocks (RTC) |
| **PLL (Phase-Locked Loop)** | Multiplies / divides another source | Generates higher or different frequencies | CPU clock when running fast |

The PLL is interesting: it takes an input clock (often the XTAL) and **multiplies or divides** the frequency. So a 40 MHz crystal can drive a 160 MHz CPU clock through a PLL ×4. PLLs also handle phase alignment so multiple clock domains stay synchronized.

```
   ┌──────────────────────────┐
   │ External 40 MHz Crystal  │  ←── 2 dedicated XTAL pins
   └─────────────┬────────────┘
                 │
                 ▼
          ┌──────────────┐
          │     PLL      │  ── multiplies / divides / phase-aligns
          │   (×4, ÷2)   │
          └──────┬───────┘
                 │
                 ▼
          160 MHz output  ── feeds CPU, peripherals, ...
```

The ESP32-C3 has all of these: an internal 17.5 MHz RC oscillator, an internal 136 kHz RC oscillator (low-power), an external 40 MHz crystal, and a PLL. The CPU clock can run from any of these depending on power vs. performance trade-offs.

## 2.9.3 — The clock tree (distribution)

Once you have a clock source, you need to **route it** to every block on the chip. The structure that does this is called the **clock tree** (or clock distribution network).

The clock tree:
- Routes the clock from its source to every block that needs it
- Includes **dividers** (scale frequency down) and **multiplexers** (select among possible sources)
- Ensures all blocks see a synchronized signal with minimal **skew** (timing offset between branches)

The book's Figure 2.15 shows the ESP32-C3 clock tree. On the left: the four sources (PLL, RC_FAST 17.5 MHz, XTAL 40 MHz, RC 136 kHz, XTAL32K 32 kHz). In the middle: a network of muxes and dividers. On the right: the destinations — `CPU_CLK`, `APB_CLK` (for peripherals), `LOW_POWER_CLK` (for sleep mode), `LEDC_SCLK` (for PWM), `CRYPTO_CLK`, `RTC_FAST_CLK`, `RTC_SLOW_CLK`.

```
   Sources                         Distribution                Destinations
   ┌─────────┐                                                ┌──────────┐
   │  PLL    │ ──────► ┌─────┐                          ┌───► │   CPU    │
   └─────────┘         │ MUX │                          │     └──────────┘
   ┌─────────┐ ──────► │     │ ─► ┌─DIV─┐ ─► ┌─MUX─┐ ───┤
   │ RC_FAST │         └─────┘    └─────┘    └─────┘    │     ┌──────────┐
   └─────────┘                                          ├───► │ Peripher.│
   ┌─────────┐                                          │     └──────────┘
   │  XTAL   │ ─► ┌─DIV─┐ ─► ┌─MUX─┐ ────────────────── ┘
   └─────────┘    └─────┘    └─────┘                          ┌──────────┐
   ┌─────────┐                                          ┌───► │   RTC    │
   │   RC    │ ────────────────────────────────────────  │    └──────────┘
   └─────────┘                                           │
                                                         ...
```

Each leaf of the tree is a clock domain — a separate part of the chip running at its own (possibly different) frequency.

## 2.9.4 — Three capabilities

Modern clock trees give software three knobs:

### Clock Division

A **clock divider** is a programmable circuit that takes an input clock and outputs a slower one (e.g. divide by 2, 4, 8). Useful when:
- A peripheral can't handle the full source frequency
- You want to save power by running the CPU at a lower rate during idle periods

### Clock Gating

A **clock gate** is a switch that can disable the clock to a part of the chip. When clock gated, the part doesn't toggle — and since power consumption in CMOS logic is largely proportional to switching activity, the part draws much less power.

Default behavior on most MCUs: **clocks to unused peripherals are gated off**. You explicitly enable a peripheral's clock before configuring it. Forgetting this is a classic embedded bug — see [2.5 — Pin Interface](./2.5-pin-interface.md#254--clock-gating-and-power).

### Clock Domains

A **clock domain** is a region of the chip running at a specific frequency, independent of other regions. Different domains let you:
- Run the CPU fast for compute, while peripherals run slower
- Keep an always-on RTC ticking on a low-power clock while the rest sleeps
- Have wireless (Wi-Fi/BLE) on a dedicated high-speed domain

When signals cross domains, special **clock domain crossing** logic is needed to avoid metastability. The MCU manufacturer handles this for you — but it explains why some peripherals have rigid clock-source rules.

## 2.9.5 — Why this matters when programming

If you ignore the clock setup, you'll hit at least one of these:

| Symptom | Cause |
|---|---|
| Peripheral seems dead, registers won't update | Clock to that peripheral is gated off |
| UART baud rate is wrong (garbage characters) | UART clock source frequency assumed wrong |
| ADC sample rate not what you expected | ADC clock divider misconfigured |
| Overall system feels slow | CPU running on RC oscillator instead of PLL |
| Crashes when entering low-power mode | Some peripheral was using a clock that gets disabled in sleep |

Knowing **which clock drives which block, and at what frequency**, is mandatory for any non-trivial peripheral configuration. The reference manual lists this for every peripheral.

---

## 2.9.6 — Quick Reference

| Concept | What it is |
|---|---|
| **Clock source** | The origin of the timing signal — XTAL, RC oscillator, PLL |
| **Crystal oscillator** | External crystal-based, accurate, stable |
| **RC oscillator** | Internal, cheap, less accurate |
| **PLL** | Multiplies / divides another source's frequency, with phase alignment |
| **Clock tree** | Distribution network from sources to all blocks |
| **Clock divider** | Programmable frequency reducer |
| **Clock gate** | Switch that disables clock to a block (saves power) |
| **Clock domain** | Region of the chip on its own clock |
| **Clock skew** | Timing offset between branches of the clock tree |

**Mental shortcut:** if you're configuring a peripheral and nothing happens, check **(a) is its clock enabled** and **(b) what frequency is it actually running at**. Both questions answered by the reference manual's clock chapter.
