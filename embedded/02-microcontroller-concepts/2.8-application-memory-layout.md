# Part 2.8 вҖ” Application Memory Layout

When the compiler and linker produce an embedded binary, they don't just spit out a blob of code. They organize it into **distinct sections**, each with a specific purpose: code, read-only data, initialized variables, uninitialized variables, and runtime regions (heap, stack). Each section ends up in a specific spot in either Flash or SRAM.

Understanding this layout is foundational вҖ” it's what determines how much memory your program uses, how fast it boots, and where bugs like stack overflow happen.

---

## 2.8.1 вҖ” The Analogy

Think of a printed cookbook on the shelf. The cookbook has:
- **Recipes** (the techniques вҖ” never change)
- **Ingredient lists** (constants вҖ” don't change)
- **Tables of pre-measured ingredients** (initialized values вҖ” known ahead)

When you start cooking, you copy the recipe steps onto a notepad (working memory), the ingredients to your **kitchen counter** (working memory), and you start adding scratch notes as you go (variables created at runtime).

The cookbook = **Flash** (read-only, persists, slow).
The kitchen counter = **SRAM** (working space, volatile, fast).

The startup of an embedded application is exactly this: copy what's needed from the cookbook (Flash) to the counter (SRAM), then start cooking.

---

## 2.8.2 вҖ” The seven memory sections

When you compile an embedded program, the toolchain reports something like:

```
   app  :
   section          size      addr
   .vector_table    1024      0x0
   .text             92      0x400
   .rodata           0      0x45c
   .data             0      0x20000000
   .bss              0      0x20000000
```

Each of these is a distinct **section**, each with its own purpose:

| Section | Contains | Volatile? | Where |
|---|---|---|---|
| **`.vector_table`** | The Interrupt Vector Table (IVT) вҖ” addresses of every ISR | No (in code) | Flash, low addresses |
| **`.text`** | Program code (functions, methods) | No | Flash |
| **`.rodata`** | Read-only data (string literals, `const` values) | No | Flash |
| **`.data`** | Initialized global variables (non-zero initial values) | Yes (mutable at runtime) | Flash вҶ’ copied to SRAM at startup |
| **`.bss`** | Uninitialized globals + globals with initial value zero | Yes | SRAM (zeroed at startup) |
| **`.heap`** | Dynamically allocated memory (`Box`, `Vec`, вҖҰ) | Yes | SRAM, grows up |
| **`.stack`** | Function call frames, local variables, ISR context | Yes | SRAM, grows down |

The runtime layout in SRAM looks like this (vertical, top is highest address):

The `.text`, `.rodata`, `.data` (initial values), and IVT all live **statically in Flash**. The rest (`.bss`, heap, stack, plus the active copy of `.data`) live in **SRAM**.

---

## 2.8.3 вҖ” Where each section lives, exactly

Two memories, three regions of life for each section:

| Section | At rest (powered off) | At runtime |
|---|---|---|
| **`.text` (code)** | Flash | Flash (executed in place) |
| **`.rodata` (constants)** | Flash | Flash (read in place) |
| **`.data` (initial values)** | Flash | Flash *(originals)* + SRAM *(working copy)* |
| **`.bss` (zero-init globals)** | nothing | SRAM (zeroed at boot) |
| **`.heap`** | nothing | SRAM (grows during execution) |
| **`.stack`** | nothing | SRAM (grows during execution) |
| **IVT** | Flash | Flash (read by CPU when interrupts fire) |

Counter to a PC вҖ” where the OS loads the entire executable into RAM and runs it from there вҖ” an embedded MCU **runs code directly from Flash**. The CPU fetches instructions from `.text` in Flash on every cycle. Only the data sections need a working copy in SRAM.

## 2.8.4 вҖ” The startup copy: Flash вҶ’ SRAM

Three things happen at boot:

1. **Copy `.data` from Flash to SRAM.** Initial values of mutable globals exist in Flash as part of the image; they need a writable home in SRAM.
2. **Zero out `.bss` in SRAM.** Globals initialized to zero (or not initialized in source) just need their SRAM region cleared.
3. **Set up the stack pointer.** The CPU's stack pointer is initialized to the top of the stack region.

After these steps, the CPU jumps to `main()`. Now the program runs вҖ” `.text` is read from Flash, `.rodata` is read from Flash, `.data` / `.bss` are read / written in SRAM, the stack grows on every function call, the heap grows on every `Box::new` or `Vec::push`.

## 2.8.5 вҖ” Practical caveats

| Caveat | Why it matters |
|---|---|
| **Stack overflow is silent on most MCUs** вҖ” when the stack grows past its limit, it overwrites the heap (or `.bss`), often corrupting state in non-obvious ways. | Estimate stack usage early. Some HALs (and `esp-hal`) provide stack-painting tools to detect overflow. |
| **The heap is optional in `no_std`**. Many embedded apps don't use a heap at all. | If your app is `no_std` without a global allocator, no `Box`, no `Vec`. Use fixed-size arrays and the `heapless` crate instead. |
| **`.bss` size doesn't show in Flash** but takes SRAM. A program with huge zero-initialized arrays has a small Flash footprint and a big SRAM footprint. | Read both Flash and SRAM size from the toolchain output. |
| **Constants belong in `.rodata`, not `.data`**. In Rust, `const FOO: u32 = 42;` ends up in `.rodata`; `static mut BAR: u32 = 42;` ends up in `.data`. | Prefer `const` over `static mut` when the value never changes вҖ” saves SRAM. |
| **The IVT lives in Flash at static, fixed addresses** (cf. [2.6.4 вҖ” Anatomy of an Interrupt](./2.6-polling-interrupts.md#264--anatomy-of-an-interrupt--source-ivt-isr)). It's typically the very first thing in the binary. | Don't try to put the IVT in SRAM or expect it to be modifiable at runtime. |

## 2.8.6 вҖ” Quick Reference

| Section | Holds | Lives in (runtime) | Mutable? |
|---|---|---|---|
| `.vector_table` (IVT) | ISR addresses | Flash | No |
| `.text` | Program code | Flash | No |
| `.rodata` | Read-only constants | Flash | No |
| `.data` | Init globals (non-zero) | SRAM (copied from Flash at boot) | Yes |
| `.bss` | Init globals (zero) | SRAM (zeroed at boot) | Yes |
| `.heap` | Dynamic allocations | SRAM | Yes |
| `.stack` | Call frames + locals | SRAM (grows downward) | Yes |

**Mental shortcut:** if it's code or unchanging data, it lives in Flash. If it's mutable at runtime, it lives in SRAM. The startup routine bridges the two by copying `.data` and zeroing `.bss`.
