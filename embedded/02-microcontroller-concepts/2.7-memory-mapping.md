# Part 2.7 — Memory Mapping

In a microcontroller, **everything has an address**. Not just memory — the peripheral registers, the I/O blocks, the system control registers, all of it. The CPU communicates with each of these by reading from or writing to its assigned address. This unification is called **memory mapping**.

It's the answer to *"how does the CPU send data to the timer? to the UART? to a memory cell?"* — same way every time: by reading or writing an address.

---

## 2.7.1 — The Analogy

Imagine a city where every building has a **postal address**. The post office (CPU) doesn't need to know what's inside each building — it just needs the address. To send a package to a bakery, it writes "123 Bakery Street" on the box. To check what arrived at a warehouse, it reads from "456 Warehouse Avenue".

A memory map is the city's master address book: every memory cell, every peripheral register, every block has its own unique street number.

The CPU never asks *"what is at this address?"*. It asks *"deliver to this address"* or *"pick up from this address"*. The hardware behind the address takes care of the rest.

---

## 2.7.2 — The general idea

When the CPU executes an instruction like *"read from address `0x4002_0010`"*, that read travels on the system bus. **Whatever block is wired to that address range responds** — it could be:
- A memory cell (returns the byte stored there)
- A peripheral register (returns the current peripheral state, e.g. ADC conversion result)
- A reserved region (returns zero or undefined)

Same goes for writes: the CPU writes to an address, the appropriate block accepts the data and acts on it.

Memory addresses are written in **hexadecimal**, with the prefix `0x` followed by hex digits. Example: `0x4002_0010` (the underscore is just a thousands separator for readability — it's not part of the value).

```
   ┌──────────────┐
   │ CPU Core     │
   │              │
   │ "Read 0x100" │
   └──────┬───────┘
          │
       System Bus
          │
   ┌──────┴───────────────────────────────────────────┐
   │                                                  │
   │  ┌────────────┐  ┌─────────┐  ┌───────────────┐  │
   │  │ Memory     │  │  Timer  │  │ Other Periph. │  │
   │  │ 0x000–     │  │  0x201– │  │ 0x301–        │  │
   │  │ 0x200      │  │  0x300  │  │ 0x3FF         │  │
   │  └────────────┘  └─────────┘  └───────────────┘  │
   └──────────────────────────────────────────────────┘
```

Each block "listens" on its assigned range. Address `0x100` lands in the Memory block. Address `0x22F` lands in the Timer's register space. The block decodes the lower bits of the address to figure out *which* cell or register is being referenced.

---

## 2.7.3 — A simple example

Suppose the memory map of a tiny MCU is:

| Address range | Block | Notes |
|---|---|---|
| `0x000 – 0x200` | Memory (code + data) | Holds the program and variables |
| `0x201 – 0x300` | Timer peripheral | Configuration and status registers |
| `0x301 – 0x3FF` | Other peripherals | ADC, GPIO, etc. |

If the CPU wants to **fetch the next instruction**, it reads from somewhere in `0x000 – 0x200` (say `0x100`). The Memory block responds with the bytes at offset `0x100`.

If the CPU wants to **configure the timer** to fire every 10 ms, it writes a specific value to a specific timer register, say `0x22F`. The Timer block decodes `0x22F` as "register #47" and updates its internal state.

The CPU never explicitly says "talk to the timer" — it just writes to an address, and the timer happens to be the block listening at that address.

## 2.7.4 — The ESP32-C3 memory map

Real chips have far more complex maps than the toy example above. The book's Figure 2.11 shows the ESP32-C3's full layout. Some highlights:

| Property | Detail |
|---|---|
| **Bus structure** | Separate **data bus** and **instruction bus** → Harvard architecture (cf. [2.3 — Memory Architecture](./2.3-processor-architectures.md#234--memory-architecture-von-neumann-vs-harvard)) |
| **ROM (Flash) — data side** | `0x3FF0_0000 – 0x3FF1_FFFF` (128 KB) |
| **ROM (Flash) — instruction side** | `0x4000_0000 – 0x4005_FFFF` (384 KB) |
| **RAM (SRAM) — data side** | `0x3FC8_0000 – 0x3FCD_FFFF` (384 KB) |
| **RAM (SRAM) — instruction side** | `0x4037_C000 – 0x403D_FFFF` (400 KB) |
| **Peripherals** | `0x6000_0000 – 0x600D_0FFF` (836 KB) |
| **Reserved regions** | Scattered throughout |

Several things to notice:

- **Same physical memory mapped at two address ranges** — ROM and RAM each appear via two ranges, one on the data bus and one on the instruction bus. This is what makes the chip Harvard: the CPU can fetch an instruction from `0x4000_xxxx` AND read data from `0x3FC8_xxxx` in the same cycle.
- **Peripherals get their own range** at `0x6000_0000`. Every register of every peripheral is somewhere inside this 836 KB block.
- **Reserved regions exist**. Reading from them is undefined; don't go there.
- **External memory** (off-chip Flash via SPI) is mapped at `0x3C00_0000` (data) and `0x4200_0000` (instruction) — the cache and MMU sit between the CPU and external memory to make this fast.

The full memory map lives in the **ESP32-C3 Technical Reference Manual** — that's the ground truth when writing low-level code.

## 2.7.5 — Memory-mapped I/O in practice

Because peripherals are addressable like memory, you can interact with them using exactly the same mechanism as reading / writing a variable. In a low-level Rust HAL, this might look like:

```rust
// Read the ADC conversion result register
let adc_val = unsafe { core::ptr::read_volatile(0x6004_0000 as *const u32) };

// Write to the GPIO output register to set pin 5 high
unsafe { core::ptr::write_volatile(0x6000_4004 as *mut u32, 1 << 5) };
```

The `volatile` qualifier matters: it tells the compiler "don't optimize this access away" — the value at that address can change due to hardware activity, even if no other code is touching it. Without `volatile`, the compiler might cache a stale read in a register and never re-fetch.

In practice, you don't typically write code like this — the HAL crate (e.g. `esp-hal`) wraps these pointer accesses in safe Rust APIs. But under the hood, every peripheral interaction is a memory-mapped read or write.

---

## 2.7.6 — Quick Reference

| Concept | What it means |
|---|---|
| **Memory map** | The assignment of address ranges to memory + peripheral blocks |
| **Memory-mapped I/O** | Peripherals are accessed by reading / writing addresses, just like memory |
| **Hex notation** | Addresses written as `0x4002_0010` (underscore is just a separator) |
| **Volatile access** | Tells the compiler "this address can change unexpectedly" — required for memory-mapped I/O |
| **Reserved range** | An address space with no block listening — don't access |
| **Reference manual** | The authoritative source for a chip's memory map |

**Mental shortcut:** every peripheral has an address. To talk to it, write / read from its address. The CPU instruction set has no special "talk to peripheral" opcodes — just `LOAD` and `STORE`.
