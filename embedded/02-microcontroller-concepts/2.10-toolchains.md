# Part 2.10 — Toolchains

A "toolchain" is the **chain of tools** that turns your source code into a binary running on a microcontroller, and the chain that lets you debug it once it's there. Embedded development has three pieces to know:

1. **Bare-metal vs RTOS** — the architectural choice for *what runs* on the chip.
2. **The compile toolchain** — how source becomes a binary on your host PC.
3. **The debug toolchain** — how that binary gets onto the MCU and how you peek inside while it runs.

---

## 2.10.1 — The Analogy

Think of a watchmaker's workshop.

- **The compile toolchain** is the assembly line: raw materials in (source code), finished watch out (binary). Each station does one thing — saw to size, polish, fit gears, calibrate.
- **The debug toolchain** is the inspection station after the watch is installed in someone's wrist: a special probe touches the back of the watch, lets you see the gears moving, set breakpoints in time, freeze and examine.
- **Bare-metal vs RTOS** is the question of *how the watch keeps time*: a single mechanical movement (bare-metal — direct, fast, hard to extend) vs an electronic movement with a microcontroller scheduling sub-modules (RTOS — flexible, more overhead, easier to add features).

Each tool has a specific shape and place on the bench. Knowing them in order saves a lot of *"why doesn't my code run on the chip?"* frustration.

## 2.10.2 — Bare Metal vs RTOS

Two approaches to building an embedded application — they differ in **what sits between your code and the hardware**.

### Bare metal

Your code runs directly on the CPU. There is no operating system. You write `main()`, configure the peripherals, set up interrupts, and the chip executes your instructions in order until power-off (or until an interrupt fires).

```
   ┌─────────────────────────────┐
   │  Application Code           │
   └─────────────┬───────────────┘
                 │ direct
                 ▼
   ┌─────────────────────────────┐
   │  CPU Core   │  Peripherals  │
   │  (Hardware)                 │
   └─────────────────────────────┘
```

- ✓ **Maximum control**, **minimum overhead**
- ✓ Fastest possible response to events
- ✓ Smallest possible binary
- ✗ No multi-tasking framework — you build it yourself if you need it
- ✗ More code to write for non-trivial systems

The Hiari book (and this wiki's `embedded/` domain) is fundamentally about **bare-metal Rust on ESP**.

### RTOS (Real-Time Operating System)

A small operating system kernel sits between your application and the hardware. It provides task scheduling (multiple "tasks" running concurrently with priorities), inter-task communication (queues, semaphores), often memory management and networking.

```
   ┌─────────────────────────────┐
   │  Application Code           │
   └─────────────┬───────────────┘
                 │
                 ▼
   ┌─────────────────────────────┐
   │  RTOS                       │
   │  ┌────────────┬─────────┐   │
   │  │ OS Kernel  │ Drivers │   │
   │  └────────────┴─────────┘   │
   └─────────────┬───────────────┘
                 │
                 ▼
   ┌─────────────────────────────┐
   │  CPU Core   │  Peripherals  │
   └─────────────────────────────┘
```

- ✓ **Multi-tasking** with deterministic scheduling
- ✓ Easier to add complex features (networking, multiple I/O streams)
- ✓ Reusable abstractions across drivers
- ✗ Memory and CPU **overhead**
- ✗ Less direct hardware control
- ✗ Determinism depends on scheduler design

Common embedded RTOSes: **FreeRTOS** (most popular), **Zephyr**, **embassy** (an `async` Rust framework that fills a similar niche).

### When to pick which

| Situation | Pick |
|---|---|
| Tight timing, simple application | Bare-metal |
| Multiple independent tasks, complex I/O | RTOS |
| Highest possible determinism / hardest real-time | Bare-metal |
| Networking, file system, multiple protocol stacks | RTOS |
| Minimum power consumption | Bare-metal (full control over every cycle) |
| Multi-core SoC with wireless | RTOS often justified |

General rule: **avoid OS overhead unless you genuinely need what it provides**. The book commits to bare-metal for its examples.

## 2.10.3 — The compile toolchain

**Goal:** turn your source files into a binary the MCU can execute.

```
   Application Code
     (.rs, .c, .h)
          │
          ▼
   ┌──────────────┐
   │   Compiler   │  ── language + ISA specific
   └──────┬───────┘
          │ Assembly file (.asm)
          ▼
   ┌──────────────┐
   │  Assembler   │
   └──────┬───────┘
          │ Object file (.o)             ┌──────────────────┐
          ▼                               │ Pre-compiled     │
   ┌──────────────┐  ◄─── object files ── │ libraries (.o)   │
   │   Linker     │                       └──────────────────┘
   └──────┬───────┘
          │ Binary executable
          ▼
   .elf, .axf, .hex, .bin
   (= machine code, layout assigned)
```

Three stages, each with one job:

### 1. Compiling

The compiler turns **high-level source** into **assembly**. It understands the language (Rust, C, C++) and targets a specific CPU architecture (ARM Cortex-M, RISC-V, Xtensa, …). The output is a `.asm` file (or held internally and passed straight to the assembler).

The compiler also performs **optimization** — dead code elimination, inlining, register allocation, etc. — and **syntax checking**. It does NOT compile pre-compiled libraries; it leaves placeholders the linker fills in later.

For Rust on ESP32-C3, the compiler is `rustc` with a RISC-V target (e.g. `riscv32imc-unknown-none-elf`).

### 2. Assembling

The assembler turns assembly text into **machine code** (binary instructions). The output is an **object file** (`.o` extension). One object file per source file. The object file contains:
- The actual machine instructions
- A symbol table (function names, global variables)
- Relocation info (where the linker needs to patch addresses later)

### 3. Linking

The linker combines:
- All your object files
- Pre-compiled object files from libraries (the standard library, HAL crates, the Rust runtime)

…and produces the **final binary**: a single executable file with all addresses resolved, layout decided (which section of memory each block goes into — see [2.8 — Application Memory Layout](./2.8-application-memory-layout.md)), and symbol references stitched together.

The output is typically `.elf` (Executable and Linkable Format) or one of `.axf`, `.hex`, `.bin`. The exact format depends on what the next stage (the flasher / debug toolchain) expects.

### Cross-compiling

You're developing on an Intel x86 PC, but the target chip is RISC-V. The compiler running on your PC must produce code for an architecture that's NOT its host. This is **cross-compiling**, and it requires:
- A cross-compiler (a compiler that targets a non-host ISA)
- A cross-linker (linker for the target)
- Cross-libraries (pre-compiled for the target architecture)

In Rust, all of this is handled by `cargo` once you set the target via `rustup target add riscv32imc-unknown-none-elf`. In C, you typically install a vendor toolchain (e.g. `arm-none-eabi-gcc`).

## 2.10.4 — The debug toolchain

**Goal:** get the binary onto the MCU, then peek inside while it runs.

```
   Host (Laptop / PC)                         Target (Dev Board)
   ┌─────────────────────────────┐
   │  Operating System           │
   │  ┌───────────────────────┐  │
   │  │  Debugging Software   │  │
   │  │  (e.g. GDB)           │  │
   │  └────────┬──────────────┘  │
   │           │                 │
   │  ┌────────▼──────────────┐  │
   │  │  Control Software     │  │
   │  │  (e.g. OpenOCD,       │  │
   │  │   probe-rs)           │  │
   │  └────────┬──────────────┘  │
   └───────────┼─────────────────┘
               │ USB
   ┌───────────▼─────────────────┐
   │  Hardware Probe / Adapter   │
   │  (e.g. ST-Link, ESP-Prog,   │
   │   FTDI, on-board USB-JTAG)  │
   └────────┬────────────────────┘
            │ JTAG / SWD / UART
   ┌────────▼────────────────────┐
   │  Microcontroller            │
   │  (ARM / RISC-V / Xtensa     │
   │   core + Flash + SRAM)      │
   └─────────────────────────────┘
```

Three layers, each with one role:

### 1. Hardware Probe / Adapter

The physical bridge between your PC's USB and the chip's debug pins. Translates USB commands into the chip's debug protocol (typically **JTAG**, **SWD**, or **UART**). Some examples:

| Probe | Protocol | Notes |
|---|---|---|
| **ST-Link** | SWD / JTAG | ST microcontrollers (also works with others via OpenOCD) |
| **J-Link (Segger)** | SWD / JTAG | High-end, fast, broad support |
| **ESP-Prog** | JTAG | Espressif's official ESP debugger |
| **Built-in USB-JTAG** | JTAG over USB | ESP32-C3 has this on-chip — no external probe needed |

The probe can usually do two things: **flash** (download a binary into Flash) and **debug** (live-attach to a running CPU).

### 2. Control Software

Runs on your PC. Translates between the debugging software (which speaks generic debug protocols like GDB Remote Serial) and the hardware probe (which speaks JTAG / SWD). Acts as a translator.

| Control software | Notes |
|---|---|
| **OpenOCD** (Open On-Chip Debugger) | Open-source, very broad chip + probe support. Default in many tutorials. |
| **probe-rs** | Rust-native, modern, growing ecosystem. Recommended for Rust embedded. |
| **espflash** (Espressif) | Pure flasher (no debugging) for ESP chips, simpler than OpenOCD when you only need to flash. |

### 3. Debugging Software

The user-facing debugger. You set breakpoints, step through code, inspect registers / memory / stack. Examples:
- **GDB** (GNU Debugger) — the workhorse, command-line, language-agnostic
- **VS Code with Cortex-Debug or probe-rs extensions** — graphical front-end on top of GDB / probe-rs
- **`cargo run`** with a debug runner — for many Rust embedded setups, this just-works

A typical workflow: write code → `cargo build` → `cargo run` (which calls `probe-rs` to flash and reset) → set a breakpoint → inspect → fix → repeat.

## 2.10.5 — Quick Reference

### Bare-metal vs RTOS

| | Bare-metal | RTOS |
|---|---|---|
| OS layer | None | Kernel + drivers |
| Overhead | Minimal | Higher |
| Multi-tasking | Build it yourself | Built-in |
| Best for | Tight timing, simple apps | Complex apps, multi-protocol |

### Compile toolchain

| Stage | Input | Output | Tool |
|---|---|---|---|
| Compile | source (.rs, .c) | assembly (.asm) | `rustc`, `gcc` |
| Assemble | assembly | object (.o) | `as`, integrated in compiler |
| Link | objects + libraries | executable (.elf) | `ld`, `lld` |

### Debug toolchain

| Layer | Role | Examples |
|---|---|---|
| Hardware probe | USB ↔ JTAG / SWD bridge | ST-Link, J-Link, ESP-Prog, on-chip USB-JTAG |
| Control software | PC ↔ probe translator | OpenOCD, probe-rs, espflash |
| Debugging software | User-facing debugger | GDB, VS Code extensions, `cargo run` |

**Mental shortcut:** to run code on a chip, you need (1) a way to compile for that chip's ISA, (2) a way to send the binary into Flash, (3) a way to talk to the chip while it runs. The toolchain provides all three in pieces.
