# Part 2.3 — Processor Architectures

A microcontroller's processor is governed by two foundational design choices: **what instructions it can execute** (its ISA) and **how it accesses memory** (Von-Neumann or Harvard). Both are fixed at silicon time — you don't choose them as a programmer; you choose an MCU that already has them.

---

## 2.3.1 — The Analogy

Two layers of "how the chip thinks":

- **The ISA** — the **language** the CPU understands. Just like English, French, and Spanish are different vocabularies, ARM, RISC-V, and x86 are different machine languages. Code compiled for one doesn't run on another.

- **The memory architecture** — the **road network** between the CPU and its memory. One single highway shared by everything (Von-Neumann) creates contention. Two parallel highways (Harvard) let traffic flow without conflicts.

Both are mechanical, both shape what the chip can do — but they answer different questions.

---

## 2.3.2 — Instruction Set Architecture (ISA)

The **ISA** is the hardware-software interface — the catalog of operations the CPU can execute. It defines:
- **What functions** the processor can do (Add, Multiply, Store, Branch, …)
- **The binary encoding** of each instruction (e.g. `MOV R1, #20` becomes a specific bit pattern)
- **The registers** available to instructions

Common embedded ISAs:

| ISA | Origin | Where you see it |
|---|---|---|
| **ARM Cortex-M** | ARM Holdings | STM32, Nordic nRF, NXP, … |
| **RISC-V** | Open standard | ESP32-C3 (this book's target), SiFive |
| **Xtensa** | Tensilica (Cadence) | ESP32, ESP32-S2, ESP32-S3 |
| **AVR** | Atmel/Microchip | ATmega (Arduino Uno) |
| **x86** | Intel/AMD | PCs, some embedded SBCs |

The ISA is **not portable**: code compiled for ARM doesn't run on RISC-V. The compiler must target the specific ISA of the chip you're flashing.

### The software-to-hardware stack

```
   ┌──────────────────────────────────┐
   │  High-level code                 │  ← Rust, C, C++ (.rs, .c, .h)
   │  (your application)              │
   └─────────────┬────────────────────┘
                 │ compiled by `rustc`, `gcc`, …
                 ▼
   ┌──────────────────────────────────┐
   │  Assembly                        │  ← .asm
   │  (architecture-specific)         │
   └─────────────┬────────────────────┘
                 │ assembled
                 ▼
   ┌──────────────────────────────────┐
   │  Instruction Set Architecture    │  ← MOV, ADD, STR, …
   │  (the contract)                  │
   └─────────────┬────────────────────┘
                 │ executed by
                 ▼
   ┌──────────────────────────────────┐
   │  ARM / RISC-V / Xtensa / …       │  ← physical CPU core
   │  Core (silicon)                  │     (Hardware)
   └──────────────────────────────────┘
```

The compiler must know the target ISA; otherwise the assembly it produces won't execute.

## 2.3.3 — A primer on buses (since we're about to need them)

A **bus** is a shared communication highway between components on the chip. Every bus has three groups of parallel lines:

| Line type | Carries | Example |
|---|---|---|
| **Data** | The bits being transferred | The actual byte being written |
| **Address** | Where to go | "Write to memory location 0x4002_0010" |
| **Control** | Direction + timing | "Read", "Write", clock signal, ready signal |

At any given clock cycle, **only one transfer can use the bus** — that's why bus design matters for performance. If the CPU and a peripheral both want to access memory at once, there's contention. Whoever holds the bus first wins; the other waits.

This contention is exactly the problem the next section solves.

---

## 2.3.4 — Memory Architecture: Von-Neumann vs Harvard

A microcontroller's memory holds two kinds of things:
- **Code memory** — application instructions, typically in non-volatile **Flash** (ROM-like)
- **Data memory** — runtime variables, in volatile **SRAM** (RAM-like)

Both must be reachable by the CPU. *How* they're reached defines two architectures.

### Von-Neumann — one bus, shared

```
   ┌─────┐                       ┌────────────────┐
   │ CPU │ ◄──────────────────►  │   Memory       │
   └─────┘   ONE BUS              │  (code +       │
            (code OR data,        │   data)        │
             one at a time)       └────────────────┘
```

A single bus carries both instructions and data. Simpler to wire, fewer pins, but the CPU **cannot fetch an instruction and read data at the same cycle** — they share the bus.

### Harvard — two buses, parallel

```
   ┌─────┐  ◄────code bus─────►  ┌────────┐
   │     │                        │ Code   │
   │ CPU │                        └────────┘
   │     │  ◄────data bus─────►  ┌────────┐
   └─────┘                        │ Data   │
                                  └────────┘
```

Two separate buses, each connecting the CPU to its own memory. The CPU can fetch an instruction AND read data **in the same cycle** — no contention, faster pipelined execution.

### Why most modern MCUs are Harvard

Pipelined CPUs do many things at once (fetch instr N+1 while executing N while reading data for N-1). With one bus, those overlapping accesses serialize and the pipeline stalls. Two buses keep the pipeline full.

Most modern embedded chips use **modified Harvard** — two buses, but both code and data memory are read/write (older Harvard was read-only on the code side). The "modified" qualifier has fallen out of use; today, "Harvard" usually means modified Harvard.

The ESP32-C3 is Harvard — you can see this in its memory map (separate data and instruction buses). See [2.7 — Memory Mapping](./2.7-memory-mapping.md).

## 2.3.5 — Quick Reference

| Question | Answer |
|---|---|
| What does the ISA define? | The instructions a CPU can execute, their binary encoding |
| Are ISAs portable? | No — code compiled for one ISA doesn't run on another |
| What ISAs are common in embedded? | ARM Cortex-M, RISC-V, Xtensa, AVR |
| What is a bus? | A shared communication highway with data + address + control lines |
| What does Von-Neumann mean? | One bus, shared between code and data memory |
| What does Harvard mean? | Two buses, separate paths for code and data |
| Which is faster for pipelining? | Harvard (no bus contention) |
| What is "modified Harvard"? | Harvard with read/write code memory (today's standard) |
