# Part 2.5 — Pin Interface

The pins of an MCU are its **physical contact with the outside world** — every wire, every sensor, every LED ultimately connects to one of these pins. But there's a problem: an MCU has many internal peripherals (GPIO, UART, SPI, I²C, ADC, …), and a typical chip package has only 30–40 pins. Far too few to give each peripheral its own dedicated set.

The solution: **multiplex the pins**. Each pin can serve multiple functions, and software chooses which one is active.

---

## 2.5.1 — The Analogy

Think of an MCU pin as a **wall socket in your house with a switchboard behind it**. Behind the wall, multiple electrical circuits are wired up — a phone line, a network cable, an old TV antenna. The socket can only carry one connection at a time. A switch on the wall (the multiplexer) decides which one is currently routed through the socket.

You don't add more sockets when you want a new connection. You repurpose the existing socket by flipping the switch.

That's exactly what pin multiplexing does on a microcontroller.

---

## 2.5.2 — Why pin multiplexing exists

| Constraint | Consequence |
|---|---|
| MCU packages are physically small (often <50 pins) | Can't dedicate one pin per peripheral function |
| MCUs have many peripherals (GPIO, UART, SPI, I²C, ADC, PWM, …) | Each peripheral wants pins |
| Most applications use a subset of peripherals at a time | Sharing pins is acceptable if software picks the right one |

So manufacturers wire each pin to **multiple internal peripherals**, and let software select which one drives the pin. The selection is done by writing to a configuration register.

### Example: ESP32-C3 pinout

The book's Figure 2.6 shows the ESP32-C3 pinout. Pins are color-coded by category:
- **Power** (VDD, GND)
- **Serial communication** (UART, USB, SPI)
- **Analog** (ADC-capable pins)
- **Digital** (GPIO)
- **Clock** (XTAL pins for external crystal)
- **Program/Debug/Control** (boot, reset, debug)

Some pins are single-purpose (e.g. power, debug). Most have **multiple possible functions**, including GPIO. For example, pin 27 on a typical microcontroller can be UART TX *or* a generic GPIO output — software picks which.

---

## 2.5.3 — The 3-stage pin function structure

When software sets up a pin to use a peripheral function (say, UART TX), three internal stages get configured:

```
   ┌──────────────────────────────────────────────────────────┐
   │                       Microcontroller                    │
   │                                                          │
   │   Clock Source  ──►  ┌─────────────────┐                 │
   │                      │ Function 1      │                 │
   │                      │  (e.g. GPIO)    │ ─┐              │
   │   Clock Source  ──►  ├─────────────────┤  │              │
   │                      │ Function 2      │  │  ┌────────┐  │
   │                      │  (e.g. Timer)   │ ─┼─►│  MUX   │──┼──► Pin
   │   Clock Source  ──►  ├─────────────────┤  │  │        │  │
   │                      │ Function n      │  │  └────────┘  │
   │                      │  (e.g. SPI)     │ ─┘     ▲        │
   │                      └─────────────────┘        │        │
   │                                            Function     │
   │       ◄── System Bus ──►                    Select      │
   │                                                          │
   └──────────────────────────────────────────────────────────┘
```

The three stages — each requiring software configuration:

### Stage 1 — Multiplexer selects the function

A **multiplexer** is a digital circuit that selects 1 of N inputs and routes it to a single output. The MCU has a multiplexer wired between each pin and its possible peripheral functions. Software writes to a config register to select which function reaches the pin.

This is the "switch on the wall" from the analogy.

### Stage 2 — The peripheral itself

The selected peripheral function (GPIO, timer, SPI, …) is itself configured by software:
- Direction (input / output for GPIO)
- Speed / frequency (baud rate for UART, prescaler for timer)
- Mode (e.g. SPI master vs slave)
- May need a **clock source enabled** before it works

Most peripherals have their own clock that must be turned on before use. This is one of the most common embedded bugs: configuring everything else but forgetting to enable the peripheral's clock.

### Stage 3 — Connection to the system bus

The peripheral connects to the system bus to expose its registers. The CPU reads / writes these registers to configure the peripheral, send data, and check status. This connection is built into silicon — software doesn't configure it directly, but it's what makes the whole thing programmable.

---

## 2.5.4 — Clock gating and power

A subtle but important detail: **clocks are an active source of power consumption**. Manufacturers default-disable clocks to peripherals that aren't being used (this is called *clock gating*).

So when you configure a peripheral, you usually need to:
1. **Enable the peripheral's clock** in the clock control register
2. Then configure the peripheral
3. Then route the pin to it via the multiplexer

Forgetting step 1 → peripheral seems dead, registers won't update, no signal on the pin. A common rite of passage in embedded debugging.

For more on the clock tree, see [2.9 — Clocks](./2.9-clocks.md).

---

## 2.5.5 — Quick Reference

| Concept | What it is |
|---|---|
| **Pin** | Physical conductor connecting MCU to outside world |
| **Pin function** | Which internal peripheral drives that pin (GPIO, UART, SPI, …) |
| **Multiplexer** | Digital circuit that selects 1 function out of N to route to the pin |
| **Multiplex / mux** (verb) | Configuring the multiplexer to pick a specific function |
| **Three stages** | Mux selection → Peripheral config + clock → Bus connection |
| **Clock gating** | Default-off clocks for unused peripherals (power saving) |

**Mental shortcut:** when a pin "doesn't work", check three things in order — (1) clock enabled? (2) multiplexer pointing to the right function? (3) peripheral itself configured?
