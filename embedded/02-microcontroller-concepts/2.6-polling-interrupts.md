# Part 2.6 — Polling vs Interrupts

How does a microcontroller's CPU know when something happens in the outside world (button pressed, ADC conversion done, timer expired)? Two strategies — it can **ask** (polling), or it can **be told** (interrupts). The choice has deep implications on responsiveness, CPU usage, and code complexity.

| | Polling | Interrupts |
|---|---|---|
| **Direction** | Pull (CPU asks) | Push (peripheral notifies) |
| **CPU usage** | High — checks even when nothing happens | Low — runs only when event fires |
| **Latency** | Up to one polling interval | Microseconds |
| **Code complexity** | Simple, linear flow | Higher (concurrency, race conditions) |

---

## 2.6.1 — The Analogy

Imagine you're working at a desk, and the doorbell at your front door is broken. You're expecting a delivery. Two strategies:

- **Polling** — every minute, you stop your work, walk to the door, check if anyone's there, walk back, resume. Most of the time, no one is there. Your work fragments. The delivery still arrives at an unpredictable time, so you might miss it by 59 seconds.
- **Interrupts** — a friend installs a new doorbell. You work without stopping. When the bell rings, you pause, go open the door, come back. No wasted trips, you react in seconds.

The doorbell mechanism = the **interrupt controller**: it watches the door, and only signals you when something actually happens. You can keep working at full focus until you're really needed.

---

## 2.6.2 — Polling — the "always-asking" model

In **polling**, the CPU repeatedly reads a peripheral's status register to check if something happened. It is a *pull* type of operation — the CPU pulls the information.

```
   Time
    │
    ▼
   ┌──────────────────────┐
   │ CPU asks peripheral: │
   │ "Anything happened?" │
   │  → Peripheral: "No"  │
   └──────────────────────┘
              ↓ (wait a bit, then again)
   ┌──────────────────────┐
   │ "Anything happened?" │
   │  → "No"              │
   └──────────────────────┘
              ↓
              ⋮
              ↓
   ┌──────────────────────┐
   │ "Anything happened?" │
   │  → "Yes — button     │
   │     pressed!"        │
   └──────────────────────┘
```

**Where polling falls short:**
- **Wasted CPU cycles** when events are rare (a button pressed once per hour → ~3600 useless checks per second if you poll every 1 ms).
- **Latency bounded by the polling interval** — if you check every 10 ms, your worst-case response is 10 ms.
- **Critical-event risk** — a safety system polling a collision sensor every 50 ms might miss a 30-ms event entirely. Unacceptable for airbags, motor faults, real-time control loops.

---

## 2.6.3 — Interrupts — the "ring-the-bell" model

In **interrupts**, the peripheral signals the CPU only when an event actually occurs. It is a *push* operation — the peripheral pushes the news.

```
                                     Button
   ┌──────────────┐                    │
   │              │                    ▼
   │   CPU runs   │     ┌──────────────────────┐
   │   normally   │     │   Peripheral block   │
   │              │ ◄── │   detects event      │
   │              │     │   → SIGNAL CPU       │
   └──────────────┘     └──────────────────────┘
          ↓
   ┌──────────────┐
   │  CPU pauses, │
   │  runs ISR,   │
   │  resumes     │
   └──────────────┘
```

**Why interrupts win on responsiveness** — the CPU only spends cycles on events that actually happen. Latency is microseconds (the time to save context, jump to the ISR). Power consumption drops because the CPU can sleep until an interrupt wakes it.

**The trade-off** — interrupt code is concurrent with main code. This is essentially a form of multi-threading, which brings classic problems:
- **Data races** — shared variables modified by both main and ISR without protection.
- **Debug difficulty** — the ISR can fire at any instruction boundary, making bugs non-deterministic.
- **Stack overflow risk** — ISRs may use a separate (small) stack on most MCUs.

---

## 2.6.4 — Anatomy of an Interrupt — Source, IVT, ISR

A complete interrupt mechanism has **three components** wired together:

| # | Component | What it is | Lives in |
|---|-----------|------------|----------|
| 1 | **Interrupt Source** | The peripheral event that triggered: ADC done, timer expired, GPIO edge, UART byte received | Hardware |
| 2 | **Interrupt Vector Table (IVT)** | A lookup table that maps each interrupt event → the address of its ISR in code memory | Code memory (low addresses, typically) |
| 3 | **Interrupt Service Routine (ISR)** | The function that runs to handle the event | Code memory, anywhere |

Think of the IVT as a **switchboard**. The source rings in. The IVT looks up *"where do I send this call?"* and routes the CPU to the right ISR.

```
   ┌─────────────────┐         ┌──────────────────────┐
   │ INTERRUPT       │         │  IVT  (in code mem)  │
   │ SOURCE          │         │  ┌────┬────────────┐ │
   │                 │         │  │ #0 │ Reset      │ │
   │ ▪ GPIO edge     │ ──────► │  │ #1 │ NMI        │ │
   │ ▪ Timer expiry  │  signal │  │ #2 │ Timer ISR  │ │ ──► jump to ISR address
   │ ▪ ADC done      │         │  │ #3 │ GPIO ISR   │ │
   │ ▪ UART RX       │         │  │ #4 │ ADC ISR    │ │
   │   ...           │         │  │ #N │  ...       │ │
   └─────────────────┘         │  └────┴────────────┘ │
                               └──────────────────────┘
```

## 2.6.5 — The 4-Step Interrupt Flow

When an interrupt fires (example: a button on a GPIO pin configured as interrupt source, code that counts presses living in a GPIO ISR):

```
   1. Peripheral signals interrupt to processor (or interrupt controller).
                    ↓
   2. Processor halts current code.
      Fetches ISR address from the IVT entry for this event.
                    ↓
   3. Processor executes the ISR (counts the press).
                    ↓
   4. Processor returns to the exact instruction it was halted on,
      and resumes normal execution.
```

The pause-and-resume is invisible to the main code — *almost* (see practical caveats below).

**Three practical caveats:**

| Caveat | Why it matters |
|---|---|
| Most MCUs use a dedicated **interrupt controller** between peripherals and CPU (NVIC on ARM, PLIC on RISC-V). It manages priorities and reduces CPU intervention until an ISR actually runs. | Configuration for interrupts touches both the peripheral *and* the interrupt controller registers. |
| Interrupts must be **enabled at two levels** — at the peripheral (local) AND at the CPU / interrupt controller (global). One alone is not enough. | A common bug: peripheral set up correctly but global IRQ flag still off → ISR never fires. |
| Interrupts have **priorities**. If two fire at once, the higher priority preempts — possibly even interrupting an already-running ISR. | "Nested interrupts" exist. Critical sections need careful design. |

## 2.6.6 — When to Use Which?

Standard answer in embedded: *"it depends on the application."* The decision rules:

| Application requirement | Polling fits? | Interrupts fit? |
|---|---|---|
| Simple, no time-critical events | ✓ — simpler code | ⚠ overkill |
| Responsive to async external events | ✗ too slow / wasteful | ✓ |
| Safety-critical / hard real-time | ✗ — risk of missing events | ✓ |
| Wake from low-power sleep | ✗ — CPU is asleep, can't poll | ✓ — only interrupts can wake the core |
| Many peripherals, complex orchestration | ✗ — polling each becomes a burden | ✓ — peripherals push only when relevant |

In practice, **complex embedded applications use a mix** — polling for slow-changing housekeeping state, interrupts for time-critical or rare events.

---

## 2.6.7 — Quick Reference

| | Polling | Interrupts |
|---|---|---|
| **Type** | Pull | Push |
| **Latency** | Up to polling interval | Microseconds |
| **CPU usage** | Wasteful when events are rare | Minimal |
| **Power** | High (CPU always running) | Low (sleep until event) |
| **Code complexity** | Low | Higher (concurrency) |
| **Debuggability** | Easy (linear flow) | Harder (non-deterministic firing) |
| **Sleep wake-up** | ✗ | ✓ |
| **Safety-critical** | ✗ | ✓ |

**Mental shortcut:** if the event is rare, time-critical, or asynchronous → interrupts. If checking the status is a normal step in the control loop → polling.
