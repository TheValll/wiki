# 18.3 — Shared state with `Mutex<T>`

Channels move ownership across threads. Sometimes you want the opposite: **all threads see the same value**, and they take turns writing to it. That's a `Mutex<T>` — *mutual exclusion* — only one thread at a time holds the lock and can access the inner value.

```rust
use std::sync::Mutex;

fn main() {
    let m = Mutex::new(5);

    {
        let mut num = m.lock().unwrap();   // acquire the lock
        *num = 6;
    }   // lock released here, when the guard is dropped

    println!("{:?}", m);   // Mutex { data: 6, .. }
}
```

`m.lock()` returns a `MutexGuard<T>` — a smart pointer that **acts like `&mut T`** and releases the lock when dropped. There is no manual `unlock()` — the type system enforces release via `Drop` (cf §17.3).

## 18.3.1 — The API

| Method | Returns | When it blocks |
|--------|---------|----------------|
| `m.lock()` | `LockResult<MutexGuard<T>>` | Blocks until the lock is free |
| `m.try_lock()` | `TryLockResult<MutexGuard<T>>` | Returns `Err` immediately if locked |
| `m.into_inner()` | `T` | Consumes the mutex, returns the inner value |

The `Result` wrapping exists because of **poisoning**: if a thread panics while holding the lock, the mutex is marked poisoned. Subsequent `lock()` calls return `Err`. You can recover via `.unwrap_or_else(|e| e.into_inner())` if you accept the risk that the data may be in a half-updated state.

## 18.3.2 — Why `Mutex<T>` is interior mutability

The pattern is the same as `RefCell<T>` (§17.5): the compiler sees a `&Mutex<T>` (a shared reference), but `lock()` hands you a `&mut` to the inner data. The aliasing rule still holds — only one writer at a time — but the check moved from compile time to **runtime via the lock**.

| | `RefCell<T>` | `Mutex<T>` |
|---|---|---|
| Threading | Single-threaded only | Multi-threaded |
| On violation | Panics | Blocks until free |
| Cost | Counter increment | OS lock (more expensive) |

`Mutex<T>` is the threaded counterpart of `RefCell<T>`.

## 18.3.3 — The `Mutex` alone is not enough

This won't compile:

```rust
let m = Mutex::new(0);

for _ in 0..10 {
    thread::spawn(move || {       // ERROR: m moved on first iteration
        let mut num = m.lock().unwrap();
        *num += 1;
    });
}
```

The `Mutex` itself is moved into the first thread; subsequent iterations have nothing to capture. We need **shared ownership across threads** — that's what `Arc<T>` is for (§18.4). The canonical pattern is `Arc<Mutex<T>>`.

```
┌──────────────────────────────────────────────────────────────────┐
│  Mutex = shared whiteboard with a single marker. Many people     │
│  in the room (Arc), but only the one holding the marker can      │
│  write. When they put it down (drop the guard), the next waiter  │
│  picks it up.                                                    │
└──────────────────────────────────────────────────────────────────┘
```

## 18.3.4 — `RwLock<T>`: many readers OR one writer

When reads vastly outnumber writes, `RwLock<T>` is faster than `Mutex<T>`: it allows **any number of concurrent readers** but only **one writer** (and no readers while writing).

```rust
use std::sync::RwLock;

let rw = RwLock::new(0);
let r1 = rw.read().unwrap();    // shared
let r2 = rw.read().unwrap();    // shared, parallel with r1
drop((r1, r2));
let mut w = rw.write().unwrap(); // exclusive
```

Same shape as `Mutex`, two methods instead of one. Use `Mutex` by default; reach for `RwLock` only when you've measured contention.
