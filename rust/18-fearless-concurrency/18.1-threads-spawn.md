# 18.1 — Threads with `thread::spawn`

A **thread** runs code in parallel with the rest of the program. Rust's standard library threads are **OS threads** (1:1 mapping) — same primitive as `pthread_create` in C, exposed safely.

```rust
use std::thread;
use std::time::Duration;

fn main() {
    thread::spawn(|| {
        for i in 1..10 {
            println!("hi {} from spawned", i);
            thread::sleep(Duration::from_millis(1));
        }
    });

    for i in 1..5 {
        println!("hi {} from main", i);
        thread::sleep(Duration::from_millis(1));
    }
}
```

`thread::spawn` takes a closure and runs it on a new OS thread. The output is **interleaved** and non-deterministic — that's concurrency.

There's a catch: when `main` ends, the program exits, **even if spawned threads are still running**. The spawned thread may print 0, 4, or 9 lines before being cut off.

## 18.1.1 — `JoinHandle`: waiting for a thread

`spawn` returns a `JoinHandle<T>`. Calling `.join()` blocks the current thread until the spawned thread finishes, and returns whatever the closure returned.

```rust
let handle = thread::spawn(|| {
    for i in 1..10 {
        println!("hi {} from spawned", i);
    }
});

// ... main work ...

handle.join().unwrap();   // wait for spawned thread to finish
```

`.join()` returns `Result<T, _>` — the `Err` case is when the thread *panicked*. `unwrap()` propagates that panic to the caller.

## 18.1.2 — `move`: transferring ownership into the thread

A spawned thread may outlive the function that started it. The compiler can't know how long it will run, so any captured reference would risk dangling. The fix: take ownership of captured values with `move`.

```rust
let v = vec![1, 2, 3];

let handle = thread::spawn(move || {     // `move` transfers ownership of v
    println!("{:?}", v);
});

handle.join().unwrap();
// println!("{:?}", v);   // ERROR: v was moved into the thread
```

Without `move`, the closure would capture `&v` — but the borrow checker rejects that, because nothing prevents `v` from being dropped before the thread reads it.

```
┌──────────────────────────────────────────────────────────────────┐
│  Rule of thumb: any closure passed to `thread::spawn` should     │
│  start with `move`. If you need to share data instead of moving  │
│  it, that's what Arc and channels are for (§18.2, §18.4).        │
└──────────────────────────────────────────────────────────────────┘
```

## 18.1.3 — Returning a value from a thread

```rust
let handle = thread::spawn(|| {
    (1..=100).sum::<u32>()
});

let total = handle.join().unwrap();   // 5050
```

The closure's return type becomes the `T` in `JoinHandle<T>`.
