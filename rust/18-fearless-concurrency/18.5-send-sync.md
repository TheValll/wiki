# 18.5 — `Send` and `Sync`: the marker traits behind it all

Everything in the previous sections — why `thread::spawn` accepts `Mutex` but not `RefCell`, why `Arc<T>` works across threads but `Rc<T>` doesn't, why `mpsc::Sender<T>` requires `T: Send` — comes down to **two marker traits** automatically derived by the compiler.

| Trait | Meaning |
|-------|---------|
| `Send` | A value of this type can be **transferred** to another thread |
| `Sync` | A value of this type can be **shared** between threads (i.e., `&T` is `Send`) |

A *marker trait* has no methods — implementing it is purely a label that says "this type satisfies a property the compiler can rely on."

## 18.5.1 — Auto-derivation

Every type is automatically `Send` and `Sync` **if all its fields are**. You almost never implement them by hand. The interesting types are the ones that **opt out**.

| Type | `Send` | `Sync` | Why |
|------|--------|--------|-----|
| `i32`, `String`, `Vec<T>` (where `T: Send`) | ✓ | ✓ | Plain data |
| `Rc<T>` | ✗ | ✗ | Non-atomic counter — racing increments would corrupt it |
| `RefCell<T>` | ✓ (if `T` is) | ✗ | Runtime borrow check is not thread-safe |
| `Cell<T>` | ✓ (if `T` is) | ✗ | Same reason |
| Raw pointers `*const T`, `*mut T` | ✗ | ✗ | No safety guarantees |
| `Arc<T>` | ✓ (if `T: Send + Sync`) | ✓ (if `T: Send + Sync`) | Atomic counter |
| `Mutex<T>` | ✓ (if `T: Send`) | ✓ (if `T: Send`) | OS-level lock |

Read the bottom rows carefully: `Mutex<T>` is `Sync` even though `T` only needs to be `Send`. That's because the lock guarantees that anyone reading or writing through `&Mutex<T>` actually gets exclusive access — so `T` only needs to be transferable, not concurrently accessible.

## 18.5.2 — Why `thread::spawn` requires `Send`

The signature:

```rust
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,
```

The closure `F` and the return type `T` must both be `Send` — they cross a thread boundary. The `'static` bound prevents borrowed references that the spawned thread might outlive.

This is why moving an `Rc` into a spawned thread fails to compile: `Rc<T>` is not `Send`, so the closure capturing it isn't `Send` either. The error appears at the `spawn` call, not deep inside thread internals — the public API enforces it.

## 18.5.3 — `Send` without `Sync`, and vice versa

The two traits are independent:

| | `Send` | `Sync` | Example |
|---|---|---|---|
| Both | ✓ | ✓ | `i32`, `Arc<Mutex<T>>` |
| Send only | ✓ | ✗ | `Cell<i32>` — can move it, but `&Cell<i32>` isn't safe across threads |
| Sync only | ✗ | ✓ | `MutexGuard<'_, T>` — can borrow it concurrently from one thread, but unlocks must happen on the locking thread |
| Neither | ✗ | ✗ | `Rc<T>`, raw pointers |

`Sync` implies `Send` only when the inner type is also `Send`, and vice versa for `Arc`. The pair forms a small algebra you'll rarely need to derive by hand — the compiler does it.

## 18.5.4 — When you might `unsafe impl Send/Sync` yourself

Wrapping a raw pointer (FFI, custom allocator, lock-free data structure) loses auto-derivation. If you have proof that your wrapper is actually thread-safe (a `Mutex` inside, careful invariants), you can:

```rust
unsafe impl<T> Send for MyWrapper<T> {}
unsafe impl<T> Sync for MyWrapper<T> {}
```

The `unsafe` keyword is the contract: *"compiler, I claim this is correct, take my word for it."* If you're wrong, you've created undefined behavior. This is rare in application code; reach for it only inside primitives.

```
┌──────────────────────────────────────────────────────────────────┐
│  Send and Sync are not features you "use" — they are the         │
│  contract that everything else (spawn, channels, Arc, Mutex)     │
│  reads. Almost every safe abstraction in std::sync requires      │
│  Send and/or Sync; almost every type satisfies them              │
│  automatically. The unsound types are explicitly !Send/!Sync.    │
└──────────────────────────────────────────────────────────────────┘
```
