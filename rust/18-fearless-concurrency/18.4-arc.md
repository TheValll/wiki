# 18.4 — `Arc<T>`: shared ownership across threads

`Arc<T>` (**Atomically Reference-Counted**) is the thread-safe sibling of `Rc<T>` (§17.4). Same API: `Arc::new`, `Arc::clone`, `Arc::strong_count`. Same semantics: shared, immutable access to a heap-allocated value, freed when the last clone is dropped. The only difference: the reference counter is updated with **atomic CPU instructions**, so two threads can `clone` and `drop` without racing.

```rust
use std::sync::Arc;
use std::sync::Mutex;
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Result: {}", *counter.lock().unwrap());   // 10
}
```

Every iteration:
1. `Arc::clone(&counter)` increments the atomic counter and returns a new owner — cheap, no deep copy.
2. `move` transfers that fresh owner into the new thread.
3. The thread `lock`s, mutates, drops the guard (releasing the lock), drops the `Arc` (decrementing the counter).

When `main` exits, the original `counter` drops; the count hits zero; the `Mutex` and its inner `i32` are freed.

## 18.4.1 — `Rc` vs `Arc`: same API, different cost

| | `Rc<T>` | `Arc<T>` |
|---|---|---|
| Counter type | Plain `usize` | `AtomicUsize` |
| Thread-safe | No (`!Send`) | Yes (`Send + Sync`) |
| Cost of clone/drop | Negligible | Atomic op (~10× slower, still cheap) |
| When to use | Single-threaded sharing (graphs, trees) | Multi-threaded sharing |

If the compiler refuses to send your `Rc` to another thread, swap it for `Arc`. If you don't need threads, stay on `Rc` — it's marginally faster.

## 18.4.2 — The two layers: `Arc<Mutex<T>>`

Read inside-out:

```
  Arc <  Mutex <  T  > >
   ↑       ↑        ↑
   │       │        └─ the actual data
   │       └─ exclusive access (one writer at a time)
   └─ shared ownership across threads (multiple owners)
```

- **`Mutex<T>`** alone gives mutability but no sharing — a single owner, locked.
- **`Arc<T>`** alone gives sharing but no mutability — many owners, read-only.
- **Together**: many owners, each can briefly become *the* writer.

## 18.4.3 — `Arc<RwLock<T>>` for read-heavy workloads

Same composition, swap the lock:

```rust
use std::sync::{Arc, RwLock};

let config = Arc::new(RwLock::new(Config::default()));

// Many threads can read concurrently:
let c = Arc::clone(&config);
thread::spawn(move || {
    let cfg = c.read().unwrap();   // shared lock
    println!("{:?}", *cfg);
});

// One thread writes occasionally:
let c = Arc::clone(&config);
thread::spawn(move || {
    let mut cfg = c.write().unwrap();   // exclusive lock
    cfg.update();
});
```

```
┌──────────────────────────────────────────────────────────────────┐
│  Pattern to internalize: Arc for sharing, Mutex/RwLock for       │
│  mutating. The two layers always come together when threads      │
│  need to write to the same value.                                │
└──────────────────────────────────────────────────────────────────┘
```

## 18.4.4 — Cycles and `Weak`, threaded edition

`Arc` has the same cycle problem as `Rc`. The fix is the same: `std::sync::Weak<T>` for the back-edge. Identical semantics to `std::rc::Weak` (§17.7), just thread-safe.
