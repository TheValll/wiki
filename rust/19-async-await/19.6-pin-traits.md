# 19.6 вҖ” `Pin`, `Send`, and async traits

Three details poke through async APIs and confuse newcomers: `Pin<&mut Self>` in `Future::poll`, `Send` bounds on spawned futures, and the historical mess around traits with async methods. This page is the *intuition*, not the deep dive вҖ” the deep dive is its own book.

## 19.6.1 вҖ” Why `Pin`?

When the compiler turns an `async fn` into a state machine, the resulting struct can hold **self-referential pointers** вҖ” a local variable pointing into another local variable, both stored in the same struct.

```rust
async fn ex() {
    let s = String::from("hello");
    let r = &s;             // r points into the same struct as s
    something().await;      // the future is paused вҖ” the struct sits somewhere
    println!("{}", r);
}
```

If the runtime were free to **move** that struct in memory between polls, the pointer `r` would point to the old location вҖ” undefined behavior. `Pin<&mut Self>` is the type system's way of saying *"this value's address must not change while you hold this pointer."* The future trait reflects that:

```rust
fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output>;
```

In application code you almost never construct `Pin` yourself вҖ” `Box::pin(future)`, `tokio::pin!`, and `pin_project` handle it. The mental rule:

## 19.6.2 вҖ” `Send` and async, the spawn trap

`tokio::spawn` requires the future to be `Send + 'static` вҖ” same rule as `thread::spawn` (В§18.1). The future may move to another worker thread, so every value held across an `.await` must also be `Send`.

The classic offender:

```rust
use std::rc::Rc;

async fn bad() {
    let r = Rc::new(42);          // Rc is !Send
    do_something().await;          // r is held across the await
    println!("{}", r);
}

tokio::spawn(bad());               // ERROR: future is not Send
```

The fix is one of:
- Drop the `Rc` before the `.await` (don't hold non-Send across await points),
- Replace `Rc` with `Arc`,
- Use a `current_thread` runtime where futures aren't `Send`.

The error messages are dense вҖ” they often point at an internal compiler-generated type. The cause is almost always *"some non-Send value lives across an `.await`."*

## 19.6.3 вҖ” `tokio::spawn_local` for `!Send` futures

If you genuinely need non-Send values across awaits (e.g. an `Rc` to a tree, a single-threaded GUI handle), use a `LocalSet`:

```rust
let local = tokio::task::LocalSet::new();
local.run_until(async {
    tokio::task::spawn_local(async {
        let r = Rc::new(42);
        do_something().await;
        println!("{}", r);
    }).await.unwrap();
}).await;
```

Tasks inside a `LocalSet` never leave their thread, so they don't need `Send`.

## 19.6.4 вҖ” Async traits

Until Rust 1.75 (Dec 2023), this didn't compile in stable:

```rust
trait Storage {
    async fn read(&self, key: &str) -> Vec<u8>;
}
```

Async fn returns an anonymous future type, and trait methods needed a known return type. The workaround was the `async-trait` crate, which boxed the future:

```rust
#[async_trait::async_trait]
trait Storage {
    async fn read(&self, key: &str) -> Vec<u8>;
}
```

The macro rewrote the method into `fn read(&self, ...) -> Pin<Box<dyn Future<...> + Send + '_>>` вҖ” heap-allocated future, plus a `Send` bound. Works fine, costs a `Box` per call.

**Since Rust 1.75 (AFIT вҖ” async fn in traits)**, the bare syntax works directly:

```rust
trait Storage {
    async fn read(&self, key: &str) -> Vec<u8>;   // stable!
}
```

Caveats still in 2026:
- The compiler infers the future's auto traits (`Send`, `Sync`) from the trait body. If you need to spawn the future via `tokio::spawn`, you may have to add an explicit `Send` bound вҖ” `trait_variant` or `return_position_impl_trait_in_trait` patterns help.
- `dyn Trait` with async methods (dynamic dispatch) is more limited than static dispatch вҖ” most applications stick to generic bounds (`T: Storage`).

For most application code today: write `async fn` in traits directly, fall back to `#[async_trait]` only if you hit a `dyn`-dispatch wall.

## 19.6.5 вҖ” The TL;DR card

| Symbol / situation | What it really means |
|---|---|
| `Pin<&mut Self>` | "Don't move this struct between polls вҖ” it has internal pointers." |
| `Box::pin(fut)` | Heap-allocate the future and pin it there. Common in trait objects. |
| Future is not `Send` | Some `!Send` value is alive across an `.await`. Find it and drop earlier, or use `Arc`. |
| `spawn_local` | A spawn that stays on this thread; no `Send` needed. |
| `#[async_trait]` | Old workaround for async-in-traits. Mostly unnecessary since 1.75. |
