# 19.4 вҖ” `Stream`: async iteration

A `Future` produces **one** value, eventually. A `Stream` produces **many** values, eventually. It is to `Iterator` what `Future` is to a synchronous function.

```rust
pub trait Stream {
    type Item;
    fn poll_next(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Option<Self::Item>>;
}
```

Each `poll_next` returns:
- `Poll::Ready(Some(x))` вҖ” here is a value,
- `Poll::Ready(None)` вҖ” the stream is done,
- `Poll::Pending` вҖ” no value yet, wake me up.

The `Stream` trait lives in the `futures` crate (and partially in `tokio_stream`). It is **not** in `std` yet вҖ” async iteration is one of the still-evolving corners of the language.

## 19.4.1 вҖ” Consuming a stream

The async equivalent of `for x in iter` is `while let Some(x) = stream.next().await`:

```rust
use futures::stream::StreamExt;   // brings .next(), .map(), etc.

let mut s = some_stream();

while let Some(item) = s.next().await {
    println!("{:?}", item);
}
```

Many runtimes also offer a `stream!` or async-`for` macro for prettier syntax, but the `while let` pattern always works.

## 19.4.2 вҖ” Where streams come from

| Source | Produces |
|--------|----------|
| `tokio::sync::mpsc::Receiver` | Stream of messages (with `tokio_stream::wrappers::ReceiverStream`) |
| `tokio::time::interval` | Stream of `Instant`s, one per period |
| Network listener (`TcpListener::accept` in a loop) | Stream of incoming connections |
| Database driver (`sqlx::query(..).fetch(..)`) | Stream of rows |
| File line iterator (`tokio::io::BufReader::lines`) | Stream of lines |

Streams are everywhere I/O is вҖ” anywhere a synchronous program would loop over `Iterator`.

## 19.4.3 вҖ” Adapter combinators

Stream has the same combinator vocabulary as `Iterator`, all defined in `StreamExt`:

```rust
use futures::stream::{self, StreamExt};

let total: u32 = stream::iter(1..=10)
    .map(|n| n * 2)
    .filter(|n| futures::future::ready(*n % 3 == 0))
    .fold(0, |acc, n| async move { acc + n })
    .await;
```

Note two async-flavored quirks:
- `filter` takes a closure returning a **future** (`futures::future::ready` lifts a sync value),
- `fold` takes an async closure (or returns a future).

Both are async because at every step you might want to do I/O. If you don't, lift sync values with `ready(...)`.

## 19.4.4 вҖ” Backpressure: streams as flow control

A stream that polls slower than its source produces **backpressure**: the source is naturally throttled because no one is asking for the next value. This is the async equivalent of a synchronous iterator that no one consumes вҖ” nothing happens. `Stream` makes that property explicit and composable, which is why it's the building block for async pipelines.

## 19.4.5 вҖ” Status as of 2026

- The `Stream` trait is **not stable in `std`** yet. Use `futures::Stream` or `tokio_stream`.
- Async iteration syntax (`for x in stream` directly) is not stable. The `for await` loop is being worked on but not in stable Rust.
- `async-stream` crate provides a `stream!` macro that lets you write a stream as if it were an `async fn` with `yield`.

```rust
use async_stream::stream;

let s = stream! {
    for i in 0..3 {
        yield i;
        tokio::time::sleep(Duration::from_millis(100)).await;
    }
};
```
