# 19.2 — Executors and runtimes (Tokio, smol, async-std)

A future is just a state machine. Something has to actually **call `poll`** in a loop, watch for I/O readiness, and reschedule futures when they wake up. That something is an **executor** — and Rust's standard library does **not** provide one.

This is unusual. In Go, Python, JS, the runtime is built into the language. In Rust, you pick your runtime as a library.

## 19.2.1 — Why no built-in runtime?

Rust targets everything from web servers to bare-metal microcontrollers. A web server wants a multi-threaded work-stealing scheduler with epoll/kqueue/IOCP. A microcontroller wants a 200-byte single-threaded loop. Bundling either would alienate the other. So `std` defines `Future` and `.await`, and lets the ecosystem provide schedulers.

## 19.2.2 — The three popular runtimes

| Runtime | Best for | Notes |
|---------|----------|-------|
| **Tokio** | Production I/O servers, ecosystem default | Multi-threaded work-stealing scheduler; integrates with `hyper`, `tonic`, `axum`, `reqwest`, `sqlx` |
| **smol** | Lightweight async, embedded-ish | Small dependency footprint, simpler internals — good for understanding |
| **async-std** | std-mirroring API | Less active than Tokio nowadays; consider Tokio first for new code |

Pick **Tokio** unless you have a reason not to. Most async crates assume Tokio.

## 19.2.3 — Tokio: the canonical entry point

```rust
#[tokio::main]
async fn main() {
    let result = fetch_data().await;
    println!("{}", result);
}
```

The `#[tokio::main]` macro rewrites your `async fn main` into:

```rust
fn main() {
    tokio::runtime::Runtime::new()
        .unwrap()
        .block_on(async { /* original body */ });
}
```

So the actual entry point is still `fn main`. The macro:
1. Builds a runtime (the executor + reactor).
2. Hands it the async block.
3. **Blocks** the OS thread on `block_on` until the future completes.

For libraries, prefer `block_on` directly — don't force runtime choices on your callers.

## 19.2.4 — What's inside a runtime

| Component | Job |
|-----------|-----|
| **Executor** | Polls ready futures; manages a work queue per worker thread |
| **Reactor** | Watches OS events (epoll/kqueue/IOCP); wakes futures when their I/O is ready |
| **Timer wheel** | Wakes futures sleeping on `tokio::time::sleep` |
| **Task spawner** | `tokio::spawn` puts a future on the queue |

The reactor talks to the OS in the language the OS understands; the executor talks to your futures. The handoff happens via `Waker` — a callback the runtime hands each `poll` so the future can say *"wake me when X happens"*.

## 19.2.5 — `tokio::spawn` vs `.await`

```rust
// Sequential: do A, then do B.
let a = task_a().await;
let b = task_b().await;

// Concurrent: start B while A is in flight.
let handle = tokio::spawn(task_b());
let a = task_a().await;
let b = handle.await.unwrap();    // JoinHandle, like thread::spawn
```

`tokio::spawn` is the async analog of `thread::spawn`: it hands a future to the runtime and returns a `JoinHandle`. Crucial difference: spawned tasks are **not** OS threads — they're cooperative tasks scheduled onto a small pool of worker threads. You can spawn 100 000 tasks; you cannot spawn 100 000 threads.

## 19.2.6 — Single-threaded vs multi-threaded runtimes

| Tokio runtime flavor | Workers | When |
|---------------------|---------|------|
| `current_thread` | 1 (the calling thread) | Tests, low-overhead servers, async on a single core |
| `multi_thread` (default) | N (CPU count) | Production servers handling many concurrent requests |

```rust
#[tokio::main(flavor = "current_thread")]
async fn main() { ... }
```
