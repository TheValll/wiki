# 19.5 — Async vs threads: when to pick which

Both threads and async let you do many things at once. They solve **different** problems. Picking the wrong one is the most common async mistake.

## 19.5.1 — The shape of the workload

The decision hinges on what your program spends time doing.

| Workload | Picture | Best tool |
|----------|---------|-----------|
| **CPU-bound** | Thread runs hot, no waiting (matrix mul, image filter, hashing, parsing big buffers) | **Threads** (`thread::spawn`, `rayon`) |
| **I/O-bound** | Thread waits on syscalls 99 % of the time (HTTP, DB, sockets, file reads, sleep) | **Async** (Tokio) |
| **Mixed** | Some I/O, occasional CPU spikes | Async runtime + `spawn_blocking` for the CPU spikes |

A web server serving 10 000 simultaneous TCP connections is the canonical I/O-bound case. A ray tracer is the canonical CPU-bound case. They want completely different tools.

## 19.5.2 — Why threads are wrong for 10k connections

Each OS thread costs ~2 MB of stack and a kernel object. 10 000 threads ≈ 20 GB of address space + huge scheduling overhead. Thread context switches go through the kernel.

A **task** (async future) is a struct with a tiny state machine — a few hundred bytes typical. Switching tasks is a function call. A single multi-threaded Tokio runtime running on 8 cores can hold a million tasks comfortably.

## 19.5.3 — Why async is wrong for matrix multiply

Async only helps when there's something to **wait on**. A `for i in 0..n { for j in 0..n { ... } }` never yields — it just runs flat-out. Wrapping it in `async` adds the state-machine overhead with no benefit. Worse: it blocks the worker thread, freezing every other task on it.

For CPU-bound work, use threads (or `rayon` for data parallelism). If the CPU work happens *inside* an async program, isolate it with `tokio::task::spawn_blocking`.

## 19.5.4 — Side-by-side: same problem, two tools

**Read 100 files concurrently.**

Threads:
```rust
use std::thread;

let handles: Vec<_> = paths.into_iter().map(|p| {
    thread::spawn(move || std::fs::read_to_string(p))
}).collect();

let contents: Vec<_> = handles.into_iter().map(|h| h.join().unwrap()).collect();
```

Async (Tokio):
```rust
use tokio::fs;
use futures::future::join_all;

let futs: Vec<_> = paths.into_iter().map(|p| fs::read_to_string(p)).collect();
let contents: Vec<_> = join_all(futs).await.into_iter().collect();
```

100 files: both work. 10 000 files: threads run out of memory; async coasts. **The crossover is around the number of concurrent in-flight operations.**

## 19.5.5 — Mixing them

Async ↔ thread is not all-or-nothing.

| Need | Tool |
|------|------|
| Small CPU burst inside async | `tokio::task::spawn_blocking` |
| Communicate from sync thread → async task | `tokio::sync::mpsc` (the sync side has a `blocking_send`) |
| Communicate from async task → sync thread | `std::sync::mpsc` (the async side does the send) |
| Run both | A Tokio runtime + `rayon`-managed worker pool, plumbed via channels |

Real systems often mix: an async server (HTTP/gRPC) hands incoming requests off to a CPU pool for the actual computation, then awaits the result.
