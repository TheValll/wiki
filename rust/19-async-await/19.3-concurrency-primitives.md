# 19.3 — Async concurrency: `join!`, `select!`, `spawn`

In synchronous code, you call functions one after the other. In async, you have a choice: **sequential** (await each in turn) or **concurrent** (start them together, then collect). The primitives below are how you build the second pattern.

## 19.3.1 — `join!`: wait for all

When you have N independent futures and you want **all** their results:

```rust
use tokio::join;

async fn fetch_all() -> (User, Posts, Avatar) {
    let (u, p, a) = join!(
        fetch_user(42),
        fetch_posts(42),
        fetch_avatar(42),
    );
    (u, p, a)
}
```

`join!` runs every future *concurrently on the same task* — they share one stack frame, no spawning, no thread-hop. The macro is roughly equivalent to manually `poll`ing each future in a loop until all are `Ready`.

If any future is slow, the total time is the **slowest** one. If all three take 100ms, the total is 100ms, not 300ms.

`tokio::try_join!` is the same but short-circuits on the first `Err`.

## 19.3.2 — `select!`: wait for the first

When you want whichever future finishes first, and you're done as soon as one does:

```rust
use tokio::select;
use tokio::time::{sleep, Duration};

async fn timeout_request() -> Result<Response, &'static str> {
    select! {
        resp = http_get("https://example.com") => Ok(resp),
        _    = sleep(Duration::from_secs(5))  => Err("timed out"),
    }
}
```

`select!` polls all branches in parallel, completes as soon as one branch's future is `Ready`, and **drops the others** mid-flight. This is how you implement timeouts, race-fastest-mirror, cancellation patterns.

| | `join!` | `select!` |
|---|---|---|
| Completes when | All branches done | Any one branch done |
| Other branches | Run to completion | Cancelled (dropped) |
| Use case | Parallel fetches, batch work | Timeouts, racing, cancellation |

## 19.3.3 — `spawn`: a real concurrent task

`join!` and `select!` interleave futures *within one task*. `tokio::spawn` creates a **separate task** the runtime can move to a different worker thread:

```rust
let handle = tokio::spawn(async move {
    process_item(item).await
});

// ... do other work ...

let result = handle.await.unwrap();
```

| Primitive | Same task / new task | Parallelism |
|-----------|----------------------|-------------|
| `.await` | Sequential, same task | None (task waits) |
| `join!` / `select!` | Concurrent, same task | None (one task) |
| `tokio::spawn` | New task | Yes (multi-thread runtime) |

Use `spawn` when:
- The work is CPU-bound enough to benefit from a different worker thread,
- You want fire-and-forget,
- Or the future's lifetime is decoupled from the caller.

## 19.3.4 — `spawn_blocking` for synchronous work

If you must call a blocking API (file I/O without async support, a CPU-heavy computation, a C library), do **not** call it directly inside `async`. It would freeze the worker thread, blocking every other task scheduled on it.

```rust
let result = tokio::task::spawn_blocking(|| {
    expensive_sync_computation()
}).await.unwrap();
```

`spawn_blocking` runs the closure on a dedicated **blocking thread pool** (separate from the worker pool) and returns a future you can `.await` — the worker stays free.

## 19.3.5 — Channels in async land

`std::sync::mpsc` blocks on `recv()` — useless inside async, you'd freeze the worker. Use the runtime's async channels:

| Channel | API |
|---------|-----|
| `tokio::sync::mpsc` | Multi-producer single-consumer, bounded or unbounded |
| `tokio::sync::oneshot` | Single value, single shot — perfect for "fire async, await result" |
| `tokio::sync::broadcast` | Multi-producer multi-consumer, every receiver gets every value |
| `tokio::sync::watch` | Single producer, multi consumer, only the *latest* value is observable |

```rust
use tokio::sync::mpsc;

let (tx, mut rx) = mpsc::channel(32);

tokio::spawn(async move {
    tx.send("hello").await.unwrap();
});

while let Some(msg) = rx.recv().await {
    println!("{}", msg);
}
```

`.send().await` and `.recv().await` are the async-friendly forms — they yield instead of blocking when the buffer is full / empty.

```
┌──────────────────────────────────────────────────────────────────┐
│  Three concurrency dials in async:                               │
│   • join!    = "all of these together, same kitchen"             │
│   • select!  = "first one to finish wins, cancel the rest"       │
│   • spawn    = "fire this off on its own line, I'll check back"  │
└──────────────────────────────────────────────────────────────────┘
```
